## PR #16 Review

Recommendation: changes requested.

### Critical

- None.

### High

1. `src/features/gear/hooks/useAutoReconnect.ts:30-56`, `src/features/home/screens/HomeScreen.tsx:83-85`, `src/features/settings/screens/SettingsScreen.tsx:16-19`
   Auto-reconnect state is set to `'connected'` after a successful reconnect, but nothing resets it when the user manually disconnects active gear from Settings. After `disconnectAll()`, `bikeConnected` / `hrConnected` become false, yet Home still renders `reconnectLabel('connected')`, so the UI reports "Connected" for disconnected gear and the auto-reconnect effect will never run again because it only fires from `'idle'`. This breaks both status accuracy and the retry path after a manual disconnect.

2. `src/features/gear/hooks/useGearSetup.ts:70-74`, `src/features/gear/screens/GearSetupScreen.tsx:49-55`, `src/features/devices/hooks/useBlePermission.ts:26-40`
   Permission denial currently becomes a silent no-op. `requestBlePermission()` returns `'denied'`, `startScan()` just returns, and `handleScanPress()` only shows the alert if `startScan()` throws, which it never does for this case. That means the primary setup CTA can appear broken when Bluetooth permission is denied or unsupported, directly undermining the required just-in-time permission UX.

### Medium

1. `src/features/gear/hooks/useGearSetup.ts:99-107`
   Any connection failure is mapped to `missing_ftms_service`, even for HR setup and even when the failure is unrelated to missing services. In the HR path this shows the wrong incompatibility reason, and in both paths it hides transient BLE errors behind a hardware-capability message. Users will be told the device is incompatible when the real problem could be connection instability or permission state.

2. `src/features/gear/hooks/useGearSetup.ts:111-149`
   The hook starts an 8-second timeout while awaiting the first signal, but it never clears that timer on unmount and never tears down the just-opened connection when the timeout fires. If the user leaves the setup screen or hits the no-signal path, the pending timer can still fire against an unmounted hook and the unsaved device stays connected/subscribed in the background until something else explicitly disconnects it.

### Low

1. `src/features/gear/hooks/__tests__/useAutoReconnect.test.ts:40-127`, `src/features/gear/hooks/__tests__/useGearSetup.test.ts:56-169`
   The added tests cover happy-path reconnect, validation failure, and timeout, but they miss the two regressions above: manual disconnect after a successful reconnect, and denied Bluetooth permission during scan start. That leaves the current suite green even though those user-facing cases are broken.
