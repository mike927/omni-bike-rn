# Running Omni Bike on Android

This app is iOS/Apple-Watch-first, but its **core ride experience** (BLE bike trainer + BLE
HR strap + training + history + Settings + Strava) runs on Android. Apple Watch and Apple
Health features are cleanly hidden on Android. This is the runbook for building and running it.

## Mental model

- **`android/` is CNG** — it is **gitignored** and regenerated by `expo prebuild` on every build.
  Never hand-edit it; change `app.config.ts` / config plugins instead.
- **`ios/` is committed** (partial eject) because the **Apple Watch companion** is a native
  SwiftUI target that prebuild can't regenerate. This is the one intentional asymmetry.
- **No local Android toolchain is required** — builds run on **EAS cloud**. (This machine has no
  JDK/SDK/adb/emulator, by choice.)

## Build profiles (`eas.json`)

| Profile | JS bundle | Needs Metro to run? | Artifact | Use it for |
|---|---|---|---|---|
| `development` | from Metro at runtime | **Yes** (dev client) | `.apk` | live debugging with Metro `[WC-JS]` logs + fast refresh |
| `preview` | **embedded** | **No** — install & open | `.apk` | "just works" standalone testing / sharing a build |
| `production` | embedded | No | `.aab` | Google Play submission (**can't sideload an AAB**) |

Each profile pulls EAS env vars from the environment named in its `"environment"` field.

## One-time setup

```bash
npx eas-cli login                       # Expo account: mikee927
npx eas-cli init --force                # links @mikee927/omni-bike-rn (already done;
                                        # owner + extra.eas.projectId live in app.config.ts)
```

Strava secrets — the cloud build can't read local `.env`, so push them per environment
(values stay on your machine + your EAS account):

```bash
npm run eas:env:push:dev        # → eas env:push development --path .env
npm run eas:env:push:preview    # → eas env:push preview --path .env
# verify names only (values stay hidden):
npx eas-cli env:list preview | grep -oE 'STRAVA_[A-Z_]+'
```

> `env:push` writes to the **project** env vars (your project → Environment variables), **not**
> the account-wide page — a common point of confusion.

## Build

```bash
# Standalone, install-and-open (recommended for just running it):
npm run eas:build:android       # → eas build --platform android --profile preview

# Dev client (needs Metro; gives live logs for verification):
npm run eas:build:android:dev   # → ... --profile development
```

EAS auto-generates the Android keystore on first build. When it finishes you get an install
URL/QR. **Gotcha:** EAS runs `npm ci` (strict on peer deps) — the React 19 conflict errors the
*Install dependencies* phase unless `.npmrc` contains `legacy-peer-deps=true` (it does; keep it).

## Install & run on a device

- **`preview` APK:** open the build's install page on the phone → **Install** → allow "install
  unknown apps" → open. No Metro, works offline.
- **`development` APK:** install the same way, then start Metro and connect:
  ```bash
  npx expo start --dev-client -c
  ```
  In the dev launcher, **scan the Metro QR** or enter `http://<your-mac-LAN-ip>:8081`
  (NOT `localhost` — that's the phone itself). Phone + Mac must share Wi-Fi.

## On-device verification checklist

1. App opens to **Home** with no red/white crash screen (proves the lazy-native-module fix).
2. First device scan → Android **"Nearby devices"** permission prompt → bike/strap reach **Ready**.
3. A short ride **saves to History**; **Strava** connects.
4. **No** "Apple Watch" HR option or "Apple Health" row anywhere (correctly hidden on Android).

> BLE can only be verified on a **physical phone** — emulators have no Bluetooth.

## `expo-doctor` is 17/18 — and that's correct

The lone failing check ("app config fields may not be synced in a non-CNG project") is triggered
by the **committed `ios/`** (required for the Watch app), not by `android/`. It is **not** a
build/CI failure (neither EAS build nor `ci:gate` run `expo-doctor`) and has no per-check
off-switch. Ignore it. Reaching 18/18 would mean deleting the Watch app or migrating the Watch
target to a CNG plugin (e.g. `@bacons/apple-targets`), which isn't worth it for a working,
on-device-verified Watch companion.

## Out of scope (deferred)

Health Connect sync (Android health-store analog of Apple Health), a Wear OS companion (wrist
HR / remote), and Play Store submission.
